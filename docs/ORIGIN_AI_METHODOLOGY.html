Move 01 · Brainstorm

Brainstorm the journey.

No feature list at the start. We picked a segment, walked the journey, and let the personas name themselves. Total elapsed: roughly a day of conversation with Claude before any code.

1.1 / Starting frame

Pick the hardest case in the segment, on purpose.

We did not start with "what could AI do for onboarding?" We started with Japanese MNCs expanding across 3+ jurisdictions — SMBC's defining segment globally and the most complex case in the book (multi-entity, multi-jurisdiction, multi-product). Solving for the hardest end means simpler segments fall out naturally.

  • Segment locked first, so every later decision could be tested against it
  • Hero scenario: Kaisei Manufacturing KK — Tokyo-HQ'd, ¥380B, expanding into SG / HK / UK with a USD 50M facility
  • Multi-jurisdiction → forces the UBO story · multi-entity → forces structure · credit facility → forces the memo

1.3 / Personas emerged from journey

We walked the day, then named the people.

Personas were not asserted up front — that produces archetypes. We walked through what a multi-jurisdiction onboarding actually feels like (same M&A doc uploaded five times, 40% of an RM's week on admin, three days to draft a credit memo) and the two people doing the work named themselves.

The receipts.

Two artefacts came out of brainstorm. Both are referenced by every later move.

Move 02 · Flow

Flow it out.

Two artefacts, one design decision. The journey table establishes the parallel-lanes model. The operational diagram formalises it. The four hero moments get placed where AI most visibly compresses the timeline.

2.1 / Two views, one journey

Customer view → Operational view.

The journey table in ORIGIN_DESIGN.md §4 lays out six stages, each with three columns: Customer · RM · AI. Three columns is the argument — the RM and the client move in parallel, and AI bridges them. Then the same six stages get re-rendered as an operational diagram in ORIGIN_PROCESS_FLOW.html, the view ops and risk read.

Note: the process-flow diagram still carries the older "SMBC Connect : Onboard" working name in its header. Renamed late.

Move 03 · Design

Set the design rules.

Tokens sourced not invented. Type stack with one job per family. One critical rule that does most of the work — and a CI check so the rule is mechanical, not aspirational.

3.1 / Palette — sourced, not invented

Trad Green and Fresh Green came from SMBC, not from us.

We asked Claude to surface the actual brand pairing from public sources, then built the token system around it.

TokenHexJob
Trad Green#004832Institutional dark · nav · headers · depth
Fresh Green#BFD730Reserved for AI — see slide 3.3
Paper#FAFBF7Warm off-white workspace surface
Mist#E8EDE4Borders, dividers
Signal amber / red / info#E8A317 · #C73E1D · #2A6F97Status only — never decorative
3.2 / Type stack — four families, four jobs

Every family does one thing.

FamilyRoleNotes
FrauncesDisplay, headlines, stat numeralsVariable serif · opsz + SOFT + WONK axes — declared inline at usage site (W-03 in CLAUDE.md)
Inter TightUI bodyTightened modern grotesque · slightly more character than Inter
Noto Sans JPJapanese charactersWeight 500 default · pairs cleanly with Inter Tight
IBM Plex MonoData, IDs, timestamps, eyebrowsMonospaced — used wherever something is technical

All four free on Google Fonts. The stack was chosen the same way as the palette — interrogating the brief through conversation, not picking from a preset.

3.3 / The one rule that does most of the work

Fresh Green is reserved exclusively for AI outputs and AI presence.

Not for generic accents. Not for primary buttons. Not for hover states on neutral chrome. Wherever Fresh Green appears in the product, intelligence happened — an extraction, a recommendation, a generated narrative. Where it doesn't, the system is doing ordinary work and looks ordinary.

This rule lives at the top of CLAUDE.md, marked critical, so every Claude Code session reads it before touching code.

3.4 / Mechanical enforcement

A guideline is a hope. A CI check is a contract.

01Source ruleCLAUDE.md "critical rule" line — read at every session start
02Detectionscripts/check-fresh-green.sh — scans for hex / token usage
03Enforcementfresh-green CI job · exceptions in .freshgreen-allowlist · architectural review to add

Same shape elsewhere: the build gate is npm run typecheck && npm run lint && npm run test && npm run build. Where it has known gaps (the EG-01 Tailwind class-scan failure mode, twice-confirmed), the gap itself is recorded in CLAUDE.md rather than left implicit.

3.5 / The five principles underneath the rule

Why the Fresh Green rule works at all.

PrincipleWhat it commits us to
AI is quiet, not loudFresh Green is rationed. The cockpit is calm. The AI is a well-dressed colleague, not a chatbot that shouts.
Parallel, not sequentialRM and client operate in parallel lanes. AI bridges them. Nothing blocks waiting for a human button-push unless judgment is required.
Explain the magicEvery AI decision is auditable — confidence scores, source registries, extraction logs, citations. The RM is in command.
Heritage through restraintJapanese design sensibility — whitespace, precision, considered typography. Kanji as watermark, not decoration.
Demo every momentThis is a showpiece. If an AI moment isn't visually memorable, it failed the mission.

Move 04 · Constrain

Constrain to the environment.

A prototype that ignores its host institution is a brochure. The hardest work in this move was deciding what not to build — and using CLAUDE.md to make sure those cuts stayed cut.

4.1 / Two anchor announcements

Origin sits in the gap between Fenergo and SMBC Connect.

Anchor 1 · Jan 2026

Fenergo rollout in Americas. SMBC Americas announced deployment of Fenergo as its CLM platform — "multi-year transformation … removing manual processes from our workflow."

Anchor 2 · Apr 2026

SMBC Connect launches in the US. Global transaction banking portal — "as intuitive and frictionless as the apps on our smartphones."

The gap. Both Fenergo and SMBC Connect target post-onboarding. Neither yet addresses the pre-onboarding 3–6 month wait. Origin is the missing front door.

4.2 / Build vs. buy posture

Origin is a UX layer on Fenergo, not a replacement.

LayerOwnerJob
CLM data modelFenergoStructured client data, periodic review, system-of-record
Client + RM experienceSMBC OriginAI-native front door · the missing pre-onboarding journey
Transaction bankingSMBC ConnectPost-activation · day-2 operations

Existing investment preserved. Missing layer added. We did not propose ripping out Fenergo to build something greenfield — that conversation was a non-starter, and the AI tooling helped surface why fast.

4.3 / Scope cuts — what we said no to

Every line below was a deliberate conversation, not an oversight.

In v0.1
  • Client portal · 6 stages · all screens
  • RM cockpit · pipeline · application detail
  • Four AI hero moments — wired, mocked or live
  • Locked design system + CI enforcement
  • Kaisei seed data + 6 portfolio applications
Out of v0.1 — and why
  • Real KYC / AML providers. Mocked — demo is about behaviour, not wiring
  • Authentication flows. Assume logged in — not what the demo is about
  • Compliance / Ops persona views. RM + Client only — adding doubles surface area
  • Mobile responsiveness. Desktop 1440px — phone is not the demo medium
  • Deep SMBC Connect integration. Stage 6 is a teaser screen
  • Production hardening. Prototype velocity over deploy-readiness

For each "out" line we asked Claude to argue the other side first — does this need to be in? When the answer is "no, and here's why" the line goes into the brief with reasoning attached. Future-us cannot un-decide it without confronting the why.

4.4 / How CLAUDE.md inherits all of this

Every constraint above lives in one file at the repo root.

CLAUDE.md is read by Claude Code at the start of every session. Constraints don't get re-explained to a fresh context — they're inherited. When we discover a new gap or watch item, it goes in the same file, in a known section.

SectionWhat it pins

Recap

Brainstorm → Flow → Design → Constrain.

01BrainstormPick the hardest case in the segment. Walk the journey. Personas name themselves.
02FlowTwo views. Place AI where it most visibly compresses the timeline.
03DesignSource tokens. One rule that does most of the work. Enforce in CI.
04ConstrainAnchor to existing strategy. Cut deliberately. Inherit via CLAUDE.md.

The output: a polished prototype in roughly seven working days, two people, that extends rather than contradicts SMBC's stated direction. The four artefacts in /docs plus this document are the receipts.

What transfers

The shape works for any process where the bottleneck is alignment, not engineering.

The transferable thing is not the prototype. It's the four-stage method behind it, with AI as a collaborator at each stage and the decisions captured in a place the next conversation can read.

Inside the bank, the same shape would work for credit underwriting, trade documentation, perpetual KYC operations — any corporate process where the time spent agreeing what the work is exceeds the time spent doing it.
