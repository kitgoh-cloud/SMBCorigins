## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `eslint.config.mjs → eslint runtime` | Lint rules govern what code is allowed to merge. A misconfigured config could disable security rules (e.g., a11y rules that flag `dangerouslySetInnerHTML`). |
| `tsconfig.json → tsc compiler` | TypeScript strictness governs runtime safety. Disabling strict checks lets unsafe code compile. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06 | Tampering | `eslint.config.mjs` ESLint rule bypass | mitigate | The config spreads `eslint-config-next/core-web-vitals` (which includes a11y + react-hooks rules) and `eslint-config-next/typescript`. PR reviewers should flag any commit that adds `eslint-disable` comments without explanation. Phase 2 does not add a `--max-warnings 0` enforcement, but Plan 02-05's CI lint job fails on any error, which is sufficient for this prototype. |
| T-02-07 | Tampering | `tsconfig.json` strictness reduction | mitigate | Both `strict: true` and `noUncheckedIndexedAccess: true` are required by D-05. Plan 02-05 CI's `typecheck` job runs `tsc --noEmit` which fails on any type error. A PR that loosened tsconfig would surface immediately as broken typecheck. |
| T-02-08 | Information Disclosure | `.gitignore` env-var leak | mitigate | `.env*.local` and `.env` patterns added in Task 2. RESEARCH §13 confirms these are project-specific additions on top of create-next-app's defaults. D-35 establishes that env vars live in Vercel dashboard, NOT in repo. |
| T-02-09 | Tampering | `.prettierrc` formatter mismatch | accept | A misformatted PR is annoying but not a security issue. The Prettier baseline is the convention; mismatch produces a `prettier --check` failure but not a bypass of any control. |

All threats LOW severity for a stakeholder demo prototype. No HIGH-severity threats introduced.
