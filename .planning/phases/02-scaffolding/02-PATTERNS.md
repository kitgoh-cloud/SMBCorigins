### `.github/workflows/ci.yml` (CI workflow / event-driven)

**Authority:** `02-RESEARCH.md` §"`.github/workflows/ci.yml` — TWO PARALLEL JOBS" + DECISIONS.md D-08, D-39.

**What to copy from (`02-RESEARCH.md` lines 564–595):**

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  typecheck:
    name: typecheck
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version-file: '.nvmrc'
          cache: 'npm'
      - run: npm ci
      - run: npm run typecheck

  lint:
    name: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version-file: '.nvmrc'
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
```

**Critical structural decisions:**
- **Two top-level jobs (NOT a matrix).** Each job emits a distinct GitHub status check named `typecheck` and `lint`. Branch protection targets each by name. A matrix would surface multiple runs of one check name, which can't be required individually (`02-RESEARCH.md` §"`.github/workflows/ci.yml`", line 599–602).
- **`actions/setup-node@v4` not v6** — v4 is current stable as of Apr 2026 (RESEARCH §"`.github/workflows/ci.yml`", line 603).
- **`node-version-file: '.nvmrc'`** auto-reads the `.nvmrc` Phase 2 creates.
- **`cache: 'npm'`** uses GitHub's npm cache layer.

**Branch protection (manual, GitHub web UI per `02-RESEARCH.md` §12 lines 627–639; `gh` CLI not installed locally):**
1. Settings → Branches → Add rule → branch pattern `main`.
2. Require PR before merging (no required reviewer count — D-08 + CODEOWNERS).
3. Require status checks: `typecheck`, `lint`, `Vercel`.
4. **Sequence:** push CI workflow first → wait for one run → THEN add protection rule (check names not auto-discoverable until first run, `02-RESEARCH.md` §"Landmines" #6).

**Pattern this establishes for Phase 3+:** Future CI jobs (e.g., a `test` job in Phase 3) get added as a third top-level job (NOT a matrix step). Each job is one required status check.

### Pattern A: CSS-first design tokens via `@theme {}`

**Source file (after Phase 2):** `app/globals.css`
**Established by:** D-26, D-27, D-29.
**Apply to:** All future style work in any phase. Token additions (e.g., shadow tokens, animation tokens) land in this same file's `@theme {}` block. Never create `tailwind.config.ts` or `styles/tokens.css`.

**Idiom:**
```css
@theme {
  --color-{name}: {hex};   /* generates bg-{name}, text-{name}, border-{name}, fill-{name}, stroke-{name} */
  --spacing-{N}: {N}px;    /* generates p-{N}, m-{N}, gap-{N}, w-{N}, h-{N} */
```

### Pattern B: next/font/google → CSS variable → `@theme inline {}` → utility

**Source file (after Phase 2):** `app/layout.tsx` + `app/globals.css`
**Established by:** D-30, D-33.
**Apply to:** Any future font additions. Never load fonts via `<link>` or `@font-face`.

**Idiom (per font):**
1. Import: `import { FontName } from 'next/font/google'`
2. Configure: `const fontVar = FontName({ subsets: ['latin'], variable: '--font-{slug}', display: 'swap' })`
3. Apply: add `${fontVar.variable}` to `<html>` className in `app/layout.tsx`.
4. Remap (in `globals.css` `@theme inline {}`): `--font-{slot}: var(--font-{slug});`
5. Use: `<p className="font-{slot}">...</p>`

**Why `@theme inline {}` and not `@theme {}`:** without `inline`, the indirection chain (`font-display utility` → `var(--font-display)` token → `var(--font-fraunces)` actual font var) breaks in Shadow DOM and certain CSS isolations. With `inline`, Tailwind generates `font-family: var(--font-fraunces)` directly. (`02-RESEARCH.md` §"Landmines" #3.)
