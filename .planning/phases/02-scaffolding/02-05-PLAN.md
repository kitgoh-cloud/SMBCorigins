    **Critical structural decisions (RESEARCH §11 lines 599–602; Pattern D):**
    - **TWO TOP-LEVEL JOBS** — NOT a matrix. Each `jobs.{key}` block becomes its own GitHub status check named after `name:`. Branch protection (Task 4) targets each by name. A matrix would surface multiple runs of one check name, which can't be required individually.
    - **`name: typecheck` and `name: lint`** explicit at the top of each job — this is what GitHub displays as the check name; without `name:`, GitHub falls back to the job key, which works but is brittle.
    - **`actions/checkout@v4` and `actions/setup-node@v4`** — RESEARCH §11 line 603 confirms v4 (NOT v6) is current stable on GitHub Actions Marketplace as of Apr 2026.
    - **`node-version-file: '.nvmrc'`** — single source of truth for Node version (D-02). Reads `.nvmrc` from Plan 02-02 (`24`).
    - **`cache: 'npm'`** — built-in npm cache layer; speeds up CI by ~30s per run.
    - **`npm ci` (NOT `npm install`)** — deterministic install from package-lock.json; fails on lockfile drift, which is exactly the safety property we want.

    **`permissions: contents: read` rationale** (security threat model T-02-17):
    - Default GitHub Actions GITHUB_TOKEN has read+write to many scopes — overpermissioned for a typecheck/lint job that only needs to read source.
    - Setting `contents: read` at workflow root grants minimum scope; both jobs inherit. This is the GitHub-recommended hardening (https://docs.github.com/en/actions/security-guides/automatic-token-authentication).
    - If Phase 3+ adds a job that needs to comment on PRs or push, narrow `pull-requests: write` per-job, not at root.

    **Do NOT use `pull_request_target` trigger** — that would run untrusted PR code with elevated permissions (the standard supply-chain attack vector documented at https://securitylab.github.com/research/github-actions-preventing-pwn-requests/). Plain `pull_request` runs PR code in the fork's permissions context — safe.

    **Do NOT add a third job for tests** — D-40 explicitly defers test framework to Phase 3. RESEARCH §"State of the Art" projects a Phase 3 `test` job will land as a third top-level `jobs.test` block.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `Pull request code → CI runner` | PR diffs run inside GitHub Actions runners with the workflow's GITHUB_TOKEN. Untrusted PR code with elevated permissions is the classic supply-chain vector. |
| `GitHub Actions Marketplace → CI runner` | Third-party actions execute with the workflow's permissions. Pinned versions reduce risk of malicious update. |
| `Vercel Bot → repo` | Vercel GitHub App reads PR commits and deploys; reports check status back. |
| `Branch protection → merge button` | Branch-protection config is the gatekeeper for what reaches `main`. Misconfiguration silently lets bad code through. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-17 | Elevation of Privilege | GITHUB_TOKEN scope | mitigate | Workflow root explicitly sets `permissions: contents: read` (Task 1 action). Default GITHUB_TOKEN otherwise has read+write to many scopes (issues, deployments, etc.). Locking to `contents: read` means a compromised CI step (e.g., a malicious npm package executing during `npm ci`) cannot use the token to push code, comment on issues, or modify deployments. https://docs.github.com/en/actions/security-guides/automatic-token-authentication |
| T-02-18 | Tampering | `pull_request_target` trigger | mitigate | Workflow uses `pull_request` (NOT `pull_request_target`). The latter would run untrusted PR code with the BASE branch's secrets and write permissions — the classic "preventing pwn requests" attack. Verify confirms `pull_request_target` does NOT appear in workflow file. https://securitylab.github.com/research/github-actions-preventing-pwn-requests/ |
| T-02-19 | Tampering | Third-party action versions | mitigate | Both actions pinned to major version: `actions/checkout@v4` and `actions/setup-node@v4`. Major-version pinning is the GitHub-recommended balance (gets security patches, locks out major breaking changes). For higher assurance, pin to commit SHA — not warranted for stakeholder demo prototype scope. |
| T-02-20 | Information Disclosure | NEXT_PUBLIC_USE_MOCK exposure | accept | The `NEXT_PUBLIC_*` prefix intentionally exposes the value to the browser bundle (Next.js convention). Value is `true` (boolean, no PII). Already accepted in Plan 02-01 T-02-03; documented here for the production-deploy boundary. **Critical for Phase 3+:** Supabase service-role keys MUST NOT use `NEXT_PUBLIC_*` prefix; only the URL and anon key may. |
| T-02-21 | Tampering | Branch protection bypass | mitigate | Task 4 configures three required status checks (typecheck, lint, Vercel) on `main`. A future PR that fails any check cannot merge. "Require PR before merging" prevents direct push to main. RESEARCH §"Landmines" #6 sequencing prevents the dropdown-empty trap. |
| T-02-22 | Information Disclosure | Source map exposure in production build | accept | Next.js 16 default produces production source maps for dev tooling. Demo prototype scope; no proprietary algorithms. If future need to disable, set `productionBrowserSourceMaps: false` in next.config.ts — not Phase 2 scope. |
| T-02-23 | Tampering | CLAUDE.md unintended modification | mitigate | Task 2 uses Edit tool (NOT Write) for CLAUDE.md, scoped to the single `_TBD_` line. Task 1 of Plan 02-01 already established the byte-identical-after-scaffold property; Task 2 here preserves it post-edit. |
